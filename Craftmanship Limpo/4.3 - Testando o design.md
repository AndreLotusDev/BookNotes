# Testando o banco de dados

Voce não testa o banco de dados, simples assim, o banco de dados geralmente se for um de confianca maxima, voce nao testa a funcionalidade dele.
Alguem ja fez isso, ainda mais se for um banco importante de mercado, como por exemplo o postgresql.

Se for testar algo, teste a queries, exemplo, situação que voce insere dois items, voce testa se apos a insercao dos dois items a query select all retorna os dois items.

Sempre desacople o banco de dados das regras de negocios.

---

Se mesmo assim voce quiser testar, como dito acima, voce deveria testar a interface que invoca esse banco de dados, pelo menos as funcionalidades que invocam essa interface e se elas funcionam da forma esperada.

# Testando GUI

Geralmente as interfaces mudam muito e são muito instáveis, você pode até testar, mas deve-se garantir que os demais modulos estejam testados em ordem de testar a interface, pois o retorno dos outros modulos em comparacao com o da GUI são ordens de grandeza de diferença.
Fora que, a GUI muda muito requerindo muito mais esforço em manutenção que as camadas mais internas.

# Testando o DESIGN

Voce ja parou pra pensar que a todo momento voce testa o codigo e garante que o design utilizado seja o mais aplicado.
Mas e testar o design por si mesmo?
Como testar que por exemplo, uma solucao em net de camada superior somente consome camadas inferiores?

Ha tambem a analise de que, um design fraco e incosistente, qualquer minima alteracao pode gerar quebras de inumeros testes, isso é um sinal de design inadequado.

Claro que ha mudancas de funcionalidades, features, infra que podem quebrar inumeros testes, mas elas como sao autoexplicativa, sao de mudancas de grandezas diferentes a pequenas features.

# Premissa de transformacao prioritaria

Página 186

Quando testamos uma aplicação e geramos testes, começamos com testes desvirtuados, os mais simples, todavia o que não percebemos é que, conforme mais fluxos tentamos cobrir, mais situações e mais flexivel tentamos tornar nosso código.
Com isso começamos com variaveis como nulo e constantes e conforme vamos iterados adicionamos if, list, iteracoes, com isso permitindo que o codigo passe em diversos fluxos diferentes.

Ou seja, passamos por uma premissa de transformacao prioritaria (TPP)

- {} -> Nil
- Nil -> Constante
- Constante -> Variavel
- Condicional -> Selecao
- Valor -> Lista
- Selecao -> Iteracao
- Instrucao -> Recursao
- Valor -> Valor Mutavel

Por mais que isso seja um caminho natural, pode ser que em certas situacoes a ordem pode mudar.
Claro que tambem, voce pode pular etapas, todavia passar essas etapas significa perder fluxos e cenarios de teste, fique muito bem atento a isso.
